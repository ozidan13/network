تحليل سيناريوهات حقيقية

3 سيناريوهات حقيقية مركّبة — كل واحد يجمع عدة Concepts. فكّر وحل قبل ما تشوف الإجابة. ده اللي بيفرّق بين اللي فاهم ومن حافظ.

Scenario 1 — Connectivity Failure

🔴 المشكلة: Hosts في VLAN 30 مش قادرين يتواصلوا مع VLAN 20. كلاهم على نفس الـ Distribution Switch. كل حاجة تانية تمام. إيه الأسباب المحتملة؟
✅ الأسباب المحتملة:
1. Inter-VLAN Routing مش مفعّل — السويتش بـ Layer-3؟ تحقق من ip routing والـ SVI للـ VLANs.
2. SVI مش موجودة أو Down — show interface vlan 20 وshow interface vlan 30.
3. ACL بتحجب — show ip access-lists وشوف لو في Rule بيرفض.
4. VLANs مش موجودة في الـ VLAN Database — show vlan brief.

Scenario 2 — DNS & Internet Failure

🔴 المشكلة: موظفين مش قادرين يفتحوا مواقع بالـ Name (google.com)، لكن بالـ IP (8.8.8.8) بيشتغل. ما المشكلة؟
✅ التشخيص:
الـ IP connectivity شغّال (ping 8.8.8.8 OK) — المشكلة في الـ DNS فقط.
1. تحقق من الـ DNS Server في الـ DHCP Options.
2. جرّب nslookup google.com — هل بيرجع جواب؟
3. لو الـ DNS Server داخلي — تحقق إن الـ DNS Service شغّال.
4. لو الـ DNS خارجي — تحقق من ACL بتسمح UDP Port 53 للخارج.

Scenario 3 — Security Incident

🔴 المشكلة: الـ IDS بيطلع Alert: "ARP Flood from 10.30.0.55". الـ SIEM بيظهر إن الـ Host ده بيبعت آلاف الـ ARP Replies في ثانية. إيه الإجراء؟
✅ الإجراء الفوري:
1. Isolate: أقطع الـ Port فوراً — shutdown على الـ Switch Port.
2. Identify: مين المستخدم؟ أي جهاز؟ هل Malware أو Intentional؟
3. Contain: تأكد إنه ما وصلش لـ VLANs تانية (VLAN Trunk Allowed List).
4. Investigate: Wireshark على الـ Mirror Port — شوف الـ ARP Replies.
5. Prevent: فعّل Dynamic ARP Inspection (DAI) وDHCP Snooping على كل الـ Access Ports.
هذا هجوم ARP Spoofing محتمل للـ MITM.
